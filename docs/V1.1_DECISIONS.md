# ITSWEBER Send v1.1 — Architecture decisions

> Status: locked 2026-05-02. Implementation in progress. Reasoning is captured here so it survives across implementation chats and so that auditors and contributors can follow the trade-offs.

This document is normative for v1.1. The cryptographic format is in `packages/crypto-spec/README.md`; this file explains why it looks the way it does.

## Goals

- Files larger than 5 GB MUST work end to end. The architecture aims for tens of GB; practical ceilings are documented per layer.
- Browser, server, storage and reverse proxy must never buffer a whole file in memory.
- v1.0 shares (manifest version 1) MUST stay decryptable.
- No clever crypto shortcuts. Each chunk is its own AES-GCM ciphertext with a unique random 96-bit IV.

## 1. Resumable upload protocol

**Decision: custom JSON-over-HTTPS resumable protocol, not tus.**

| Option                              | Why we picked / rejected                                                                                                                                                                                                                                                                                                                                    |
| ----------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `tus-node-server` + `tus-js-client` | Heavy. Wire format conflicts with our ciphertext-chunk model — every chunk in our world is its own AES-GCM ciphertext with metadata (IV) that has to live inside the encrypted manifest. tus assumes a single byte stream and does not give us a clean place for per-chunk metadata. The client library would also need a custom storage backend per chunk. |
| **Custom resumable**                | Fits our model directly. Per-chunk metadata (IV, length) lives in the encrypted manifest the client builds. Server-side state is small and survives crashes. Endpoint surface is narrow (5 routes).                                                                                                                                                         |

**Endpoints (`apps/api/src/routes/uploads-resumable.ts`)**

| Method   | Path                                                            | Body                                                                                                                                                | Purpose                                                                                                                                                                                     |
| -------- | --------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `POST`   | `/api/v1/uploads`                                               | JSON: `{ blobs: [{ blobId, plaintextSize?, cipherSize, chunkCount }], expiryHours, downloadLimit, passwordProtected, salt?, ivWrap?, wrappedKey? }` | Reserve an `uploadId`. Server validates quota, creates pending entry, returns `{ uploadId, chunkSize, maxBlobBytes }`.                                                                      |
| `PATCH`  | `/api/v1/uploads/:uploadId/blobs/:blobIndex/chunks/:chunkIndex` | raw bytes (`application/octet-stream`)                                                                                                              | Append a single ciphertext chunk. Idempotent on `(uploadId, blobIndex, chunkIndex)` — server rejects out-of-order writes within a blob, accepts re-PATCH of the last chunk if size matches. |
| `GET`    | `/api/v1/uploads/:uploadId`                                     | —                                                                                                                                                   | Resume state: `{ uploadId, blobs: [{ blobId, receivedChunks, receivedBytes }], expiresAt }`.                                                                                                |
| `POST`   | `/api/v1/uploads/:uploadId/finalize`                            | multipart: `manifest`, `manifest-iv`                                                                                                                | Atomically commit: convert pending upload into a real share, write manifest and meta.json, return `{ id, wordcode, expiresAt }`.                                                            |
| `DELETE` | `/api/v1/uploads/:uploadId`                                     | —                                                                                                                                                   | Cancel an in-progress upload, free server resources.                                                                                                                                        |

Auth: same auth as `/api/v1/upload` (cookie session OR bearer token OR anonymous if accounts are not required). Per-IP rate limit on `POST /uploads` (creation) and per-upload rate limit on `PATCH` enforced by Fastify's rate-limit plugin.

Pending uploads expire after `UPLOAD_RESUME_HOURS` (default 24 h). A periodic cleanup job removes expired pending uploads and their partial blobs.

The legacy single-shot `POST /api/v1/upload` (manifest v1) is kept for 1.0 client compatibility. Any new client uses the resumable path.

## 2. Manifest format v2

```jsonc
{
  "version": 2,
  "files": [
    {
      "name": "<plaintext filename>",
      "size": <plaintext size, bytes>,
      "mime": "<content-type>",
      "blobId": "blob-0001",
      "chunkSize": 16777216,            // plaintext chunk size in bytes
      "chunks": [
        { "iv": "<base64url-12-bytes>", "cipherSize": 16777232 },
        { "iv": "<base64url-12-bytes>", "cipherSize": 16777232 },
        { "iv": "<base64url-12-bytes>", "cipherSize":  4194320 }   // last chunk, smaller
      ]
    }
  ],
  "note": "optional markdown"
}
```

Each `chunks[i].cipherSize` is plaintext-bytes-in-this-chunk + 16 (GCM auth tag). The decoder reads `cipherSize` bytes from the blob stream, decrypts with `(master_key, chunks[i].iv)`, emits the plaintext chunk to the writer (file save / ZIP encoder / Web Stream).

Wire layout on storage for v2:

```
<share-id>/
  manifest          ← AES-GCM ciphertext of the manifest JSON
  manifest.iv       ← 12 bytes
  blob-0001         ← concatenation of all chunks' ciphertexts in order (no separators)
  blob-0002         ← …
  meta.json         ← unencrypted: createdAt, expiresAt, downloadLimit, passwordRequired, salt, ivWrap, wrappedKey, manifestVersion
```

v2 no longer writes per-blob `.iv` files (those were a v1-only artifact — IVs now live in the encrypted manifest, where they belong).

`meta.json` carries `manifestVersion: 2` so the recipient client can pick the right decoder before downloading the manifest. v1 shares have no `manifestVersion` field; absence implies v1.

### Why per-chunk random IV instead of derived

A 96-bit random IV with AES-GCM has an IV-reuse risk that grows with the number of distinct IVs. Even at 2³² chunks (4 billion) under the same key, the collision probability is well below 2⁻³². Each share gets a fresh `master_key`, so the only IVs ever encrypted under that key are this share's chunks (≤ a few thousand for any realistic file). Random per-chunk IV is therefore safe and removes a class of bugs (deterministic counter logic, off-by-one IV reuse on retry).

### Backwards compatibility

The download client reads `meta.json` first. If `manifestVersion === 2` → v2 decoder. Otherwise → v1 decoder. v1 decoder loads `blob-NNNN.iv` and decrypts the whole blob in one AES-GCM call (existing behaviour).

## 3. Storage adapter — stream extension

`StorageAdapter` gains stream methods alongside the existing buffer methods. Buffer methods stay for small artifacts (manifest, meta.json) where a one-shot write is cheaper than a stream.

```typescript
interface StorageAdapter {
  // existing buffer methods (kept):
  put(shareId: string, name: string, data: Buffer): Promise<void>;
  get(shareId: string, name: string): Promise<Buffer>;
  exists(shareId: string): Promise<boolean>;
  delete(shareId: string): Promise<void>;
  expireBefore(cutoff: Date): Promise<string[]>;

  // new in v1.1:
  /**
   * Append `source` to <shareId>/<name>, creating the file if it doesn't exist.
   * Used by the resumable upload path to write chunks one by one without
   * holding the whole blob in memory.
   */
  appendStream(shareId: string, name: string, source: Readable): Promise<{ bytesWritten: number }>;

  /** Total bytes currently written for <shareId>/<name>, or null if missing. */
  size(shareId: string, name: string): Promise<number | null>;

  /**
   * Stream `<shareId>/<name>` to the caller. Optional inclusive byte range.
   * Used by the download path to support Range requests and to avoid loading
   * full blobs into memory.
   */
  getStream(
    shareId: string,
    name: string,
    range?: { start: number; end?: number },
  ): Promise<Readable>;
}
```

**Filesystem implementation**: `appendStream` uses `fs.createWriteStream(path, { flags: 'a' })`; `getStream` uses `fs.createReadStream(path, { start, end })`; `size` uses `fs.stat`. Native, zero-copy.

**S3 implementation, first cut**: `appendStream` opens an S3 multipart upload on the first chunk for that `(shareId, name)`, persists the `UploadId` and ETags in `uploads_in_progress`, completes the multipart on finalize. `getStream` issues a `GetObject` with `Range: bytes=start-end` and returns the response body stream. `size` uses `HeadObject`.

S3 multipart constraints (10 000 parts, 5 MiB minimum part size except last) put a per-blob ceiling on the S3 backend. With the default 16 MiB chunk size the ceiling is 156 GB per blob via S3. The filesystem backend has no such ceiling. This is documented in `docs/LARGE_FILES.md`.

## 4. Chunk size

**Decision: 16 MiB plaintext per chunk by default**, configurable via `CHUNK_SIZE_BYTES`.

| Constraint                     | Implication                                                                                 |
| ------------------------------ | ------------------------------------------------------------------------------------------- |
| S3 multipart minimum part size | 5 MiB → 16 MiB satisfies.                                                                   |
| S3 multipart max parts         | 10 000 → 16 MiB chunk caps S3-backed blob at ~156 GB.                                       |
| Browser AES-GCM throughput     | ~100–300 MiB/s on modern HW → ~50–160 ms per chunk encrypt.                                 |
| Per-chunk RAM                  | 16 MiB plaintext + 16 MiB ciphertext = 32 MiB transient. Safe even on small mobile devices. |
| HTTP overhead                  | 16 MiB chunks → 64 round-trips per GB, well under any rate limit.                           |

The server announces the chunk size to the client in the `POST /api/v1/uploads` response so the client doesn't need to be hard-coded.

## 5. ZIP streaming for multi-file downloads (Block B — next turn)

**Decision: `client-zip` (zip64 support, native streams) plus File System Access API where available.**

- Primary path: download manifest → for each file, open a writable stream via `showSaveFilePicker` → for each blob, fetch `getStream`, decrypt chunk-by-chunk through the encoder, pipe into the ZIP encoder, pipe into the writable stream. Browser writes directly to disk.
- Fallback path (Safari, older Firefox): individual file download stays available. The "Download all as ZIP" button is hidden with an explanatory tooltip when File System Access API is missing AND total decrypted size is above an in-memory ceiling (default 1 GiB).

This is the Block B design but it is _not implemented yet_. Block B follows in the next iteration.

## 6. Quota and download-limit accounting under streaming

- Quota is decremented at upload create time (`POST /api/v1/uploads`) by the **declared** total ciphertext size, with a hard cap of `MAX_UPLOAD_SIZE_BYTES` (admin setting). On finalize the server compares the actual sum of received chunk sizes to the declared total; mismatch → 409 and the pending upload is rolled back. On cancel the quota reservation is released. This avoids a TOCTOU race where two concurrent uploads each individually fit in the quota.
- The download limit decrement happens once per recipient share-download — defined as a successful manifest fetch followed by completed blob streams. Multi-file ZIP counts as one share-download. Range-resumed downloads do not re-decrement within a short window keyed by `(shareId, recipientHash)`.

## 7. External limits — what has to be raised for >5 GB

| Layer                          | Setting                     | v1.1 default                                              |
| ------------------------------ | --------------------------- | --------------------------------------------------------- |
| Caddy reverse proxy            | `request_body { max_size }` | bumped to `64GB` in `docker/Caddyfile.example` (was 5 GB) |
| Fastify single-shot `/upload`  | multipart `fileSize`        | unchanged at 500 MB (legacy path)                         |
| Fastify resumable PATCH        | per-request body limit      | `chunkSize + 4 KiB` slack                                 |
| Docker tmpfs `/tmp`            | size                        | unchanged at 64 MiB (no upload buffering in tmpfs)        |
| Admin setting                  | `max_upload_size_bytes`     | default 100 GB; hard limit enforced server-side           |
| S3 multipart                   | parts × chunk size          | 10000 × 16 MiB = ~156 GB per blob                         |
| Browser File System Access API | `showSaveFilePicker`        | required for ZIP-streaming fallback (see Block B)         |

These are documented end-to-end in `docs/LARGE_FILES.md` (created in Block A).

## 8. JavaScript byte-counter safety

`Number.MAX_SAFE_INTEGER` is 2⁵³ − 1 (~9 PB). Per-chunk and per-blob sizes are stored and added as plain JS numbers; this is safe up to PB-scale shares. No `BigInt` is needed. Server-side accumulators use the same type. Quota and `total_size_bytes` are `INTEGER` columns in SQLite (64-bit signed, ~9 EB ceiling).

## 9. Out of scope for v1.1

- Forward secrecy across uploads (each upload still uses a fresh master key — same as v1.0).
- Per-recipient access control (the URL fragment is still the capability).
- Server-side AV scan of ciphertext (impossible by design — server only sees ciphertext).
